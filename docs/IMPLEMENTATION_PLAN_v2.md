# トマトオク v2 実装計画

- 文書種別: 将来版の実装計画
- 対象: `chameleonjp-lab/tomatooku`
- 基準: `main`（PR #3反映後）

> v2はまだ実装されていない。各回の目的を1つに絞り、完了条件を満たしてから次へ進む。

## 1. 全体方針

- `main`へ直接変更しない
- 1回の作業目的を1つにする
- 設計、実装、検証、文書更新を同じPull Request内で完結させる
- 共通Supabaseの既存関数を推測で置き換えない
- 公式ランキングの条件を揃えてから公開する
- 現行の複数ファイル・ES Modules構成を維持する
- 3MB上限や1ファイル化を必須条件にしない
- 実機で遊べることを、実装方式より優先する

## 2. 第1回：v2文書と契約の確定

### 目的

将来版の要件、技術契約、実装順序をリポジトリへ残す。

### 変更ファイル

```text
docs/REQUIREMENTS_v2.md
docs/SPEC_v2.md
docs/IMPLEMENTATION_PLAN_v2.md
README.md
package.json
```

### 変更しない範囲

```text
index.html
src/**
scripts/**
package-lock.json
docs/SUPABASE_SETUP.md
docs/TEST_REPORT.md
既存のv1文書
```

### テスト

- `git diff --check`
- `package.json`のJSON解析
- 現行テストを変更していないことの確認

### 合格条件

- v2文書に未実装と明記
- 公式3問、練習、補正タイム、共通RPCを明記
- v1とv2を混同しない
- ゲームコード差分なし

### 残リスク

文書は確定しても、ゲームはv1のまま。

### 戻し方

この文書Pull Requestを閉じる。ゲームコードへの影響はない。

## 3. 第2回：共通ランキング連携

### 目的

実験場の共有Supabase契約へ`src/ranking.js`を合わせる。

### 変更予定ファイル

```text
src/ranking.js
ランキング設定を置くファイル
ランキング単体テスト
必要なv2文書
```

### 変更しない範囲

- UIデザイン
- タイマー
- スコア式
- 問題選出
- ステージデータ

### 作業

1. 共有Supabaseの現行RPC定義を確認
2. `submit_score`の正確な引数・返却値を確定
3. `get_best_score_ranking`、`get_first_try_ranking`を実装
4. `p_display_name`、`p_game_slug`、`p_score`、`p_client_version`を送る
5. `CLIENT_VERSION`を定数化
6. `ok / empty / error / not_configured`を区別
7. `AbortController`でタイムアウト時に通信を中止
8. play ID単位の二重送信防止を実装

### テスト

- 正常送信body
- 公式モードだけ送信
- 同一play IDの二重送信防止
- 取得0件
- HTTPエラー
- タイムアウト
- 形式不正
- 未設定

### 合格条件

- 実Supabaseでテスト名の初回・ベストを確認
- 他ゲームの関数やデータを変更していない
- UIは現行のままでも通信状態を判定できる

### 残リスク

補正タイムはまだ実装されていないため、本番ランキングへは公開しない。

### 戻し方

旧`ranking.js`へ戻し、設定差分を撤回する。共有Supabaseに変更を加えた場合は別途ロールバック手順を残す。

## 4. 第3回：タイマーと非同期競合の修正

### 目的

計測開始点を正確にし、古い待機処理が新しいプレイを壊さないようにする。

### 変更予定ファイル

```text
src/game.js
src/main.js
index.html（カウントダウンの土台のみ）
scripts/game.test.js
scripts/e2e.test.js
v2文書
```

### 変更しない範囲

- ランキングの表示デザイン
- 補正タイムへの切り替え
- 公式・練習モード
- 生成器

### 作業

1. `performance.now()`へ変更
2. `home / countdown / playing / stageTransition / result / retired`を導入
3. play IDを導入
4. transition timer IDを保持
5. リタイア・再開始時に待機処理を解除
6. 盤面描画後に計測を開始
7. クリア瞬間に計測を終了
8. ステージ別時間を保持
9. 共有キャンセルの`AbortError`を通常終了として扱う

### テスト

- カウントダウン中は未計測
- 描画後に開始
- 演出時間を除外
- `finishStage`二重呼び出し
- クリア演出中のリタイア
- リタイア直後の再開始
- 旧コールバック無効化
- 共有キャンセル時にコピーしない

### 合格条件

- 例外・二重進行なし
- ステージ別時間の合計が総実プレイ時間と一致
- 現行ゲームフローを維持

### 残リスク

ランキング値はまだv1方式。

### 戻し方

状態遷移とタイマー変更を同一Pull Request単位で戻す。

## 5. 第4回：補正タイムと2モード

### 目的

公式ランキングの公平性と、練習のランダム性を両立する。

### 変更予定ファイル

```text
index.html
src/game.js
src/main.js
src/ranking.js
src/styles.css
scripts/game.test.js
scripts/e2e.test.js
v2文書
```

### 変更しない範囲

- ステージ生成器
- 大規模なデザイン刷新
- 実験場側リポジトリ

### 作業

1. 公式3問の設定構造を作る
2. ランダム練習の選出を分離
3. 有効な練習3問組を列挙して選ぶ
4. 補正タイム計算を実装
5. 公式だけを送信
6. 練習結果へ明示表示
7. 結果に実時間と加算内訳を表示
8. 公式3問のIDは人間のプレイ確認後に確定

### テスト

- 公式は毎回同じ3問・同じ順
- 難易度1・2・3
- 公式3問の正解配置非重複
- 練習のID・正解配置非重複
- 練習は送信しない
- 補正タイム計算
- 3分超の値
- 表示と送信値の一致

### 合格条件

- 公式と練習を誤認しない
- 練習結果が本番ランキングへ混ざらない
- 公式3問の条件が全プレイヤーで同じ

### 残リスク

公式3問が暗記される可能性。全員同条件を優先する。

### 戻し方

v1ランキングへ戻す場合、v2データを混在させない。必要に応じて公開停止またはslug分離を行う。

## 7. 第6回：生成器 v2

### 目的

練習用の問題数と多様性を増やし、検証不足の盤面を出さない。

### 変更予定ファイル

```text
scripts/generate_stages_v2.js
scripts/verify_stages_v2.js
新しいステージバンク
生成器テスト
v2文書
```

### 作業

1. 14種類の正解配置を列挙
2. 回転・上下反転・左右反転を正規化
3. 正解5マスを別エリアの種にする
4. 各エリアを連結した5マスへ成長
5. 一意解を全探索で確認
6. A〜E限定、ID、difficulty、solution範囲・重複を検査
7. エリア連結を独立検証
8. 対称形・近似盤面を除外
9. 14配置の偏りを抑制
10. seedとgenerator versionを保存
11. 盤面内容から安定IDを生成
12. 最低84問を目標にする

### 難易度

探索ノード数だけに依存せず、人間向け解法を模した指標を追加する。

- 候補削除
- 強制配置
- 隣接禁止による確定
- 仮定回数
- 先読み深さ

### テスト

- 厳密な形式
- 一意解
- エリア連結
- 対称形重複
- 安定ID
- seed再現性
- 解配置の分布
- 生成失敗時の上限

### 合格条件

- 全問題が独立検証に合格
- 似た盤面の量産を抑制
- 公式ランキングにはまだ自動採用しない

### 残リスク

自動難易度と人間の体感が一致しない可能性。

### 戻し方

現行30問バンクを保持し、新バンクを切り替え可能にする。
